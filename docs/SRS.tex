	\section{PoC Scope}
	The PoC validates core platform viability through eight deliverable modules:
	\begin{enumerate}
		\item Authentication with role-based access control (Administrator, Blue Team Operator, Red Team Operator, Purple Team Operator, Learner, Visitor)
		\item Centralized security dashboard with key metrics
		\item Manual log ingestion and parsing
		\item Alert visualization with severity classification
		\item ML-based anomaly scoring using Isolation Forest
		\item AI-powered assistant for alert explanation
		\item Red Team tool integration (network port scanner)
		\item File Integrity Monitoring (SHA-256 baseline and verification)
	\end{enumerate}

	\chapter{Context and Problem Statement}
	% ══════════════════════════════════════════════════════════════════════════
	
	\section{Industry Context}
	Global cybercrime costs continue to rise exponentially, yet security education and training lack affordable, integrated environments where students and professionals can practice both defensive and offensive security in a single platform.
	
	\section{Core Problem Statement}
	\begin{warningbox}[title=Central Problem]
		How to design and implement a unified, accessible cybersecurity platform that combines threat monitoring, AI-assisted analysis, and machine learning anomaly detection for academic and training purposes?
	\end{warningbox}

	\section{HELIX Positioning}
	\begin{enumerate}
		\item \textbf{Unified Platform:} Combines Blue Team monitoring, Red Team tools, and AI assistance in one interface.
		\item \textbf{AI-Native:} Integrates machine learning anomaly detection and LLM-powered analysis from the ground up.
		\item \textbf{Academic Focus:} Designed for learning, demonstration, and research with accessible architecture.
	\end{enumerate}

	\chapter{Project Objectives}
	% ══════════════════════════════════════════════════════════════════════════
	
	\section{General Objective}
	Develop a functional proof-of-concept for a modular cybersecurity platform that demonstrates log ingestion, anomaly detection, AI-assisted analysis, and role-based access control within a unified web interface.
	
	\section{Functional Objectives}
	\begin{itemize}
		\item Implement user authentication with six access levels (Administrator, Blue Team Operator, Red Team Operator, Purple Team Operator, Learner, Visitor)
		\item Build a centralized dashboard displaying security metrics and alert summaries
		\item Enable manual log file upload and structured parsing
		\item Visualize security alerts with severity-based classification
		\item Apply Isolation Forest ML model to detect anomalous log patterns
		\item Integrate OpenAI API for natural language alert explanation
		\item Implement one Red Team tool (network port scanner) with controlled access
		\item Implement File Integrity Monitoring with SHA-256 baseline and verification
	\end{itemize}
	
	\section{Academic Objectives}
	\begin{itemize}
		\item Demonstrate full software engineering lifecycle (requirements to implementation)
		\item Produce complete UML diagrams (Use Case, Class, Sequence, Component, Deployment)
		\item Showcase multi-language integration (PHP, Python, Java, JavaScript)
		\item Deliver a demonstrable portfolio piece for academic evaluation
	\end{itemize}
	
	\section{Non-Functional Objectives}
	\begin{itemize}
		\item Modular architecture supporting future MVP expansion
		\item Clean, maintainable code following established conventions
		\item Secure authentication with password hashing and session management
		\item Responsive interface functional at standard desktop resolutions
		\item Comprehensive documentation for academic submission
	\end{itemize}

	% ══════════════════════════════════════════════════════════════════════════
	
	\section{Actor Overview}
	HELIX defines one abstract generalization actor (\textbf{Authenticated User}), five concrete human actors, one unauthenticated actor, and two secondary system actors:
	
	\begin{figure}[H]
		\centering
		\begin{sectionbox}
			\begin{tabular}{C{3.2cm} C{4.0cm} C{2.8cm} C{2.2cm}}
				\toprule
				\textbf{Actor} & \textbf{Profile} & \textbf{Access Level} & \textbf{Type} \\
				\midrule
				\textit{Authenticated User} & \textit{Abstract (generalization)} & \textit{Common auth. capabilities} & --- \\
				\midrule
				Administrator & Platform manager & Full access to all modules & Primary \\
				Blue Team Operator & SOC analyst & Monitoring, alerts, AI, logs & Primary \\
				Red Team Operator & Penetration tester & Offensive tools, scans, results & Primary \\
				Purple Team Operator & Hybrid operator & Blue + Red combined view & Primary \\
				Learner & Authenticated student & Learning materials, notes, progress & Primary \\
				Visitor & Unauthenticated user & Public content, registration & Primary \\
			Detection Engine & Autonomous system & ML scoring, alert generation & Secondary \\
			AI Service & Internal system & OpenAI-powered AI responses & Secondary \\
			\bottomrule
			\end{tabular}
		\end{sectionbox}
		\caption{HELIX PoC Actor Summary}
	\end{figure}
	
	The conceptual hierarchy is:
	\texttt{Visitor} (unauthenticated); \texttt{Authenticated User} $\rightarrow$ \texttt{Administrator},
	\texttt{Blue Team Operator}, \texttt{Red Team Operator}, \texttt{Purple Team Operator}, \texttt{Learner};
	and \texttt{Purple Team Operator} inherits from both \texttt{Blue Team Operator} and \texttt{Red Team Operator}.
	Secondary system actors (\texttt{Detection Engine}, \texttt{AI Service}) interact with the system boundary externally.

	\section{Non-Functional Requirements}
	
	\subsection{Performance}
	\begin{itemize}
		\item \textbf{NFR-001}: Dashboard API endpoint (GET /dashboard/stats) SHALL respond within 500ms at p95 under 5 concurrent users on localhost (XAMPP).
		\item \textbf{NFR-002}: ML anomaly scoring SHALL complete within 5 seconds for a batch of 500 log entries on the reference hardware (8 GB RAM, Intel i5).
		\item \textbf{NFR-003}: Log file upload and parsing SHALL complete within 10 seconds for files up to 5 MB (verified by automated upload test).
		\item \textbf{NFR-004}: AI assistant response SHALL return within 15 seconds (including OpenAI API round-trip) for standard alert-context queries.
	\end{itemize}
	
	\subsection{Security}
	\begin{itemize}
		\item \textbf{NFR-010}: Passwords hashed with bcrypt (cost factor $\geq$ 10). Verified by inspecting password\_hash() calls in source.
		\item \textbf{NFR-011}: PHP sessions SHALL use HttpOnly + Secure cookie flags. Verified by integration test.
		\item \textbf{NFR-012}: All user-supplied input SHALL be validated and sanitized before use. All shell\_exec() calls SHALL use escapeshellarg(). Verified by 100\% of unit tests for input validation passing.
		\item \textbf{NFR-013}: SQL injection prevention via prepared statements (PDO). Zero raw SQL string concatenation with user input. Verified by static analysis.
		\item \textbf{NFR-014}: All authenticated endpoints SHALL return HTTP 401 for unauthenticated requests. Verified by 100\% of AuthMiddleware unit tests passing.
	\end{itemize}
	
	\subsection{Maintainability}
	\begin{itemize}
		\item \textbf{NFR-020}: Modular file structure with clear separation of concerns (Controllers/Services/Models). No PHP function SHALL exceed 40 lines of code.
		\item \textbf{NFR-021}: Consistent PSR-4 namespace and naming conventions across all PHP files. Verified by automated linting.
		\item \textbf{NFR-022}: All API endpoints SHALL be documented in \texttt{docs/SDD/API\_REFERENCE.md} with method, route, auth requirements, and role restrictions.
		\item \textbf{NFR-023}: Configuration files separated from application logic. Sensible defaults provided in \texttt{api/Config/.env.example}.
	\end{itemize}
	
	\subsection{Usability}
	\begin{itemize}
		\item \textbf{NFR-030}: Interface SHALL render correctly at 1280x720 minimum resolution without horizontal scrolling.
		\item \textbf{NFR-031}: Severity-based color coding (Critical=red, High=orange, Medium=yellow, Low=green) consistent across all views.
		\item \textbf{NFR-032}: All error responses SHALL include a human-readable message in JSON format.
		\item \textbf{NFR-033}: Navigation between modules SHALL require at most 2 clicks from the dashboard.
	\end{itemize}
	
	\subsection{Deployability}
	\begin{itemize}
		\item \textbf{NFR-040}: Runs on XAMPP (Apache 2.4 + MySQL 8 + PHP 8.x) without additional infrastructure. Verified by clean install on Windows/macOS/Linux.
		\item \textbf{NFR-041}: Python ML service runs as standalone script invoked by PHP via shell\_exec(). Output SHALL be stdout-only JSON (no direct DB writes from Python).
		\item \textbf{NFR-042}: Database schema provided as single importable SQL file at \texttt{database/schema.sql}. 
		\item \textbf{NFR-043}: Configuration via \texttt{api/Config/.env} file with \texttt{.env.example} template provided.
	\end{itemize}
