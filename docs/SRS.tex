	\section{PoC Scope}
	The PoC validates core platform viability through seven deliverable modules:
	\begin{enumerate}
		\item Authentication with role-based access control (Administrator, Blue Team Operator, Red Team Operator, Purple Team Operator, Learner, Visitor)
		\item Centralized security dashboard with key metrics
		\item Manual log ingestion and parsing
		\item Alert visualization with severity classification
		\item ML-based anomaly scoring using Isolation Forest
		\item AI-powered assistant for alert explanation
		\item One Red Team tool integration (network port scanner)
	\end{enumerate}

	\chapter{Context and Problem Statement}
	% ══════════════════════════════════════════════════════════════════════════
	
	\section{Industry Context}
	Global cybercrime costs continue to rise exponentially, yet security education and training lack affordable, integrated environments where students and professionals can practice both defensive and offensive security in a single platform.
	
	\section{Core Problem Statement}
	\begin{warningbox}[title=Central Problem]
		How to design and implement a unified, accessible cybersecurity platform that combines threat monitoring, AI-assisted analysis, and machine learning anomaly detection for academic and training purposes?
	\end{warningbox}

	\section{HELIX Positioning}
	\begin{enumerate}
		\item \textbf{Unified Platform:} Combines Blue Team monitoring, Red Team tools, and AI assistance in one interface.
		\item \textbf{AI-Native:} Integrates machine learning anomaly detection and LLM-powered analysis from the ground up.
		\item \textbf{Academic Focus:} Designed for learning, demonstration, and research with accessible architecture.
	\end{enumerate}
	
	% ══════════════════════════════════════════════════════════════════════════
	\chapter{Project Objectives}
	% ══════════════════════════════════════════════════════════════════════════
	
	\section{General Objective}
	Develop a functional proof-of-concept for a modular cybersecurity platform that demonstrates log ingestion, anomaly detection, AI-assisted analysis, and role-based access control within a unified web interface.
	
	\section{Functional Objectives}
	\begin{itemize}
		\item Implement user authentication with six access levels (Administrator, Blue Team Operator, Red Team Operator, Purple Team Operator, Learner, Visitor)
		\item Build a centralized dashboard displaying security metrics and alert summaries
		\item Enable manual log file upload and structured parsing
		\item Visualize security alerts with severity-based classification
		\item Apply Isolation Forest ML model to detect anomalous log patterns
		\item Integrate OpenAI API for natural language alert explanation
		\item Implement one Red Team tool (network port scanner) with controlled access
	\end{itemize}
	
	\section{Academic Objectives}
	\begin{itemize}
		\item Demonstrate full software engineering lifecycle (requirements to implementation)
		\item Produce complete UML diagrams (Use Case, Class, Sequence, Component, Deployment)
		\item Showcase multi-language integration (PHP, Python, Java, JavaScript)
		\item Deliver a demonstrable portfolio piece for academic evaluation
	\end{itemize}
	
	\section{Non-Functional Objectives}
	\begin{itemize}
		\item Modular architecture supporting future MVP expansion
		\item Clean, maintainable code following established conventions
		\item Secure authentication with password hashing and session management
		\item Responsive interface functional at standard desktop resolutions
		\item Comprehensive documentation for academic submission
	\end{itemize}

	\section{Non-Functional Requirements}
	
	\subsection{Performance}
	\begin{itemize}
		\item \textbf{NFR-001}: Dashboard loads within 2 seconds for up to 1,000 log entries.
		\item \textbf{NFR-002}: ML anomaly scoring completes within 5 seconds for batch of 500 entries.
		\item \textbf{NFR-003}: Log file upload and parsing completes within 10 seconds for files up to 5 MB.
		\item \textbf{NFR-004}: AI assistant response returns within 8 seconds.
	\end{itemize}
	
	\subsection{Security}
	\begin{itemize}
		\item \textbf{NFR-010}: Passwords hashed with bcrypt (cost factor $\geq$ 10).
		\item \textbf{NFR-011}: PHP sessions with secure cookie flags.
		\item \textbf{NFR-012}: Input validation and sanitization on all user inputs.
		\item \textbf{NFR-013}: SQL injection prevention via prepared statements (PDO).
		\item \textbf{NFR-014}: Role-based access enforced on every API endpoint.
	\end{itemize}
	
	\subsection{Maintainability}
	\begin{itemize}
		\item \textbf{NFR-020}: Modular file structure with clear separation of concerns.
		\item \textbf{NFR-021}: Consistent naming conventions across all languages.
		\item \textbf{NFR-022}: Documented API endpoints with input/output specifications.
		\item \textbf{NFR-023}: Configuration files separated from application logic.
	\end{itemize}
	
	\subsection{Usability}
	\begin{itemize}
		\item \textbf{NFR-030}: Interface functional at 1280x720 minimum resolution.
		\item \textbf{NFR-031}: Severity-based color coding consistent across all views.
		\item \textbf{NFR-032}: Clear error messages for failed operations.
		\item \textbf{NFR-033}: Intuitive navigation between modules.
	\end{itemize}
	
	\subsection{Deployability}
	\begin{itemize}
		\item \textbf{NFR-040}: Runs on XAMPP (Apache + MySQL + PHP) without additional infrastructure.
		\item \textbf{NFR-041}: Python ML service runs as standalone script invoked by PHP.
		\item \textbf{NFR-042}: Database schema provided as importable SQL file.
		\item \textbf{NFR-043}: Configuration via editable config files (no environment variable dependency for PoC).
	\end{itemize}
